53 Actually now I am much more interested in way smaller memories ;)
54 
55 0000 0000 Size Free bits
56 0000 0FFF 4KB 20 |
57 0000 1FFF 8KB 19 |
58 0000 3FFF 16KB 18 |
59 0000 7FFF 32KB 17 |
60 0000 FFFF 64KB 16 |
61 0001 FFFF 128KB 15 |
62 0003 FFFF 256KB 14 | - This range is very interesting.
63 0007 FFFF 512KB 13
64 000F FFFF 1MB 12
65 001F FFFF 2MB 11
66 003F FFFF 4MB 10
67 007F FFFF 8MB 9
68 00FF FFFF 16MB 8
69 01FF FFFF 32MB 7
70 03FF FFFF 64MB 6
71 07FF FFFF 128MB 5
72 0FFF FFFF 256MB 4
73 1FFF FFFF 512MB 3
74 
75 Those are the kind of platforms that are fun... so a bunch of
76 wasted bits in heap pointers if we run on small MCUs.
77 
78 -----------------
79 
80 it is also the case that not all addresses will be used if all "cells" are
81 of the same size, 8 bytes...
82 
83 value 0: 0000 0000
84 value 1: 0000 0008
85 value 3: 0000 0010
86 value 4: 0000 0018
87 
88 Means bits 0,1,2 will always be empty in a valid address.
89 
90 Cons cells also need to be have room for 2 pointers. So each allocated cell from
91 memory should be 8bytes.
92 
93 Things that needs to be represented within these bits:
94 
95  - GC MARK one per cell
96  - TYPE: type of CAR and type of cons
97 
98 Types I would want:
99  - Full 32bit integer. Does not leave room for identification of type
100  - Float values. Same problem
101 
102 
103 Free bits in pointers 64MB heap:
104 31 30 29 28 27 26 2 1 0
105 0 0 0 0 0 0 XX XXXX XXXX XXXX XXXX XXXX X 0 0 0
106 
107 
108 Information needed for each cell:
109  Meaning | bits total | bits per car | bits per cdr
110  GC mark | 2 | 1 | 1 - only one of them will be used (the other is wasted)
111  Type | 2x | x | x
112  Ptr/!ptr | 2 | 1 | 1
113 
114 
115 Types (unboxed):
116  - Symbols
117  - 28bit integer ( will need signed shift right functionality )
118  - 28bit unsigned integer
119  - Character
120 
121 If four types is all that should be possible (unboxed). then 2 bits are needed to differentiate.
122 2 + 1 + 1 = 4 => 28bits for data.
123 
124 bit 0: ptr/!ptr
125 bit 1: gc
126 bit 2-3: type (if not ptr)
127 bit 3 - 24 ptr (if ptr)
128 bit 4 - 31 value (if value)
129 
130 An unboxed value can occupy a car or cdr field in a cons cell.
131 
132 types (boxed) extra information in pointer to cell can contain information
133  - 32 bit integer
134  - 32 bit unsigned integer
135  - 32 bit float
136 
137 boxed representation:
138  [ptr| cdr]
139  |
140  [Value | Aux + GC_MARK]
141 
142 Kinds of pointers:
143  - Pointer to cons cell.
144  - Pointer to unboxed value (fixnums not in a list, I hope this is so rare that it can be removed )
145  - integer
146  - unsigned integer
147  - symbol
148  - float
149  - Pointer to boxed value.
150  - 32 bit integer
151  - 32 bit unsigned integer
152  - 32 bit float
153  - (Maybe something else ? Vectors/strings allocated in memory not occupied by heap?)
154  - vector of int
155  - vector of uint
156  - vector of float
157  - vector of double
158  - String
159 
160 13 pointer"types" -> needs 4 bits
161 for 64MB heap there are 6 free bits. So with this scheme going to 128MB or 256MB heap
162 is also possible
163 
164  a pointer to some off heap vector/string could be represented by
165 
166  [ptr | cdr]
167  |
168  [full pointer | Aux + GC_MARK]
169  |
170  [VECTOR]
171 
172 Aux bits could be used for storing vector size. Up to 30bits should be available there
173 >> This is problematic. Now the information that something is a vector is split up
174 >> between 2 cons cells. This means GC needs both of these intact to be able to make
175 >> proper decision.
176 >> Will try to resolve this by adding some special symbols. But these must be symbols
177 >> that cannot occur normally in programs. Then an array could be:
178 
179  [Full pointer | ARRAY_SYM + GC_MARK]
180  |
181  [VECTOR]
182 
183 >> Boxed values same treatment as above.
184 >> TODO: Could this be simpler?
185 
186 [ VALUE | TYPE_SYM + GC_MARK]
